S1. 인트로
(이전 영상 일부 컷 삽입)
드디어 개인 프로젝트로 구상하던 에이전트를 만들었습니다. 설계했고, 개발했고, 배포했고, ChatGPT 앱 연동까지도 확인했습니다.
이제 여러분께 공유만 하면 되는 상황이었는데, 작업을 마무리하는 순간 굉장히 크리티컬한 이슈 하나를 마주하게 됐습니다.
그래서 오늘은 “에이전트 만들기 튜토리얼”이 아니라, 에이전트를 만들면서 반드시 한 번은 부딪히게 되는 진짜 문제를 이야기해보려고 합니다.

S2. 개인 프로젝트로서의 에이전트
안녕하세요. AI LLM 개발자 크랩입니다.
팀이나 회사 단위에서는 에이전트나 MCP 기반 작업을 여러 번 해봤지만, 이번에는 정말 지극히 개인적인 사이드 프로젝트로 에이전트를 처음부터 끝까지 직접 만들어봤습니다.
설계 → 개발 → 배포 → 앱 연동 → 실사용 테스트까지. 하루에 2~3시간씩, 대략 2주 정도 걸렸습니다.
원래는 “개인 프로젝트 완성기”로 과정을 100% 공개하려고 버전 관리도 하면서 꽤 설렜는데요.
작업을 마치고 보니 과정보다 더 중요한 이야기를 먼저 해야겠다는 생각이 들었습니다. 그래서 주제를 바꿨습니다.
이 영상이 관련 업을 하시는 분들께는 조금 불편할 수도 있습니다. 누군가를 들추거나 비판하려는 의도는 전혀 없고요. 결론은 하나입니다.
에이전트를 더 잘 만들고, 더 잘 쓰기 위한 이야기입니다.

S3. 에이전트 소개 – ToolChecker
에이전트 이름은 툴체커(ToolChecker) 입니다.
이 프로젝트는 제 개인적인 욕망(?)과 구독자분들께 실제로 도움이 될 만한 가치의 교집합에서 시작했습니다.
요즘 가장 많이 받는 질문 중 하나가 이거예요.
“이럴 때 AI 도구 뭐 쓰면 좋을까요?”
사실 저도 모든 도구를 다 알고 있지는 않습니다. 아는 만큼은 솔직하게 이야기하지만, 항상 마음 한켠에는 “이 비교가 정말 충분히 검증된 걸까?”라는 답답함이 남아 있었어요.
그래서 아예 업무 상황을 입력하면 AI 도구를 비교·검증해주는 에이전트를 만들면 저뿐만 아니라 팀 단위에서도 쓸 수 있겠다는 생각으로 시작했습니다.
설계 단계에서 그렸던 시나리오는 이렇습니다.

유저: 우리 팀 자동화를 일부 도입하려고 하는데 Zapier / Make / n8n 중에 뭐가 좋아? 난 개발자 아니고, 팀에 개발자 1명만 있어.

에이전트: 목표가 다를 수 있어요. 자동화 유형이 어떤 쪽에 가까울까요? A) 반복 업무(메일, 시트, 슬랙) B) API 기반 복잡한 워크플로우 C) 온프레미스 또는 자체 호스팅 필요

유저: A + B. 그리고 데이터가 민감해서 자체 호스팅도 고려 중이야.
에이전트: 그렇다면 핵심 기준은
* 비개발자 사용성
* 자체 호스팅 가능 여부
* 확장성
입니다. 공식 문서 기준으로 기능·배포·보안 옵션을 검증하고, 상충되는 부분은 보류 처리하겠습니다.
최종 산출물은 노션 보고서 (PDF + 노션 링크) 형태로 제공합니다.

S4. 그래서… 안 된 건가요?
결론부터 말씀드리면, 반은 됐고, 반은 안 됐습니다.
설계부터 배포까지 전 과정은 정상적으로 완료됐고, ChatGPT 앱 연동을 통한 실사용도 가능합니다.
그런데 문제가 있었습니다.
여기서 하나 짚고 가야 할 개념이 있습니다.
ChatGPT, Gemini, Claude 같은 LLM은 외부 도구를 사용할 때 내부적으로 이런 판단을 합니다.
1. 이 질문에 내가 바로 답할 수 있는가?
2. 외부 도구를 쓰는 게 더 나은가?
툴체커는 구조적으로 외부 MCP / 앱 연동 도구입니다.
그래서 ChatGPT 입장에서는
* 응답 속도는 내부 모델보다 느리고
* 정보 밀도도 내부 추론 결과보다 낮다고 판단되면
**“굳이 이 도구를 쓸 필요가 없다”**고 결론을 내리고 툴체커를 아예 호출하지 않습니다.
비유하자면 이런 상황이에요.
시험 기간에 핵심만 정리한 요약 노트를 만들어놨는데, 전공서적 입장에서는 “분량도 적고, 맥락도 부족하다”고 판단해서 아예 참고하지 않는 상황입니다.

S5. 그럼 어떻게 해야 하나?
처음엔 이렇게 생각했습니다.
* 속도를 더 줄여야 하나?
* 정보량을 더 늘려야 하나?
그런데 테스트를 반복하면서 아주 중요한 인사이트를 얻었습니다.
제가 아무리 열심히 만들어도 ChatGPT 자체를 이길 수는 없습니다. 이건 너무 당연한 이야기죠.
그래서 방향을 바꿨습니다.
기능을 줄이고, 설계를 더 뾰족하게 만들고, “이 에이전트만 할 수 있는 영역”에 정보 밀도를 극단적으로 압축하자.
이 방향으로 버전 2를 만들고 다시 테스트했습니다.
그 결과, 이전에는 5번 중 1번 정도만 툴체커를 참고하던 흐름이 이후에는 5번 중 3~4번 수준으로 호출되는 비율로 올라갔습니다.
(정확히 말하면 ‘참고 빈도’가 유의미하게 증가했습니다.)

S6. 에이전트의 진짜 비밀
사실 저는 이전부터 계속 궁금했습니다.

“왜 LLM모델들은 
어떤 상황에서는 에이전트를 쓰고, 어떤 상황에서는 아예 배제할까?”

예전에는 이걸 단순히 “가챠 확률”처럼 생각했던 것 같습니다.

그런데 개인 프로젝트를 하면서 논문과 실제 테스트를 같이 보니까 이게 훨씬 명확해졌습니다.

LLM은 기본적으로 ‘결과 품질 대비 비용’으로 판단합니다.

많은 사람들이 이렇게 생각합니다.
* 에이전트를 만들면 자동으로 쓰일 거라 생각
* 기능이 많을수록 좋을 거라 생각
하지만 실제로는 이렇습니다.
* LLM은 항상 “내가 직접 하는 게 더 싸고 좋은가?”를 계산하고
* 에이전트는 전문성의 압축체여야 선택됩니다.
그래서 제가 에이전트를 강조해온 이유도 여기 있고, 
이 시점에서 제 개인 프로젝트의 50프로 실패담을 공개하는 게 오히려 도움이 되겠다고 판단했습니다.

에이전트는 기존 LLM을 대체하는 게 아닙니다.
LLM이 싫어하는 일을 대신해주는 구조여야 합니다.
그리고 하나 더 중요한 걸 배웠습니다.
에이전트는 “언제 쓰일지”만 설계하는 게 아니라, “언제 안 쓰일지”를 설계하는 게 정말 중요합니다.

저의 이번 50프로 실패담이 에이전트를 만들고 계신 분들께도, 에이전트를 더 잘 쓰고 싶은 분들께도 작은 힌트가 되었으면 좋겠습니다.

지금까지, 어렵지 않게 일에 바로 쓰는 AI. 크랩이었습니다.
